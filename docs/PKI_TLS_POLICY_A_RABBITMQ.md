# PKI scripts, policy A, and RabbitMQ (local Docker)

This document describes the **OpenSSL-based** scripts under `local-broker/pki/scripts/`, the shared **`.cnf`** files, how each certificate and key fits the project’s **CA hierarchy and issuing model (policy A)**, and how to **adjust RabbitMQ in Docker** for MQTTS. It is the operator-facing companion to the high-level [PLAN_TLS_RABBITMQ_CA_CHAIN_AND_TRUSTSTORE.md](./PLAN_TLS_RABBITMQ_CA_CHAIN_AND_TRUSTSTORE.md).

**Client truststore (`ca-bundle.pem`):** step **`06-assemble-chain-and-truststore.sh`** writes the **root CA** certificate to `local-broker/truststore/ca-bundle.pem`. The **intermediate** still signs the server leaf, and the broker still presents **leaf + intermediate**; however **OpenSSL 3 and Python** perform PKIX path building that requires the **root** in the client’s trust file to validate the intermediate. A bundle with **intermediate only** fails verification with `unable to get issuer certificate`. For a “no root on clients” operational policy in production, you would typically rely on a **corporate root already in the system trust store** or a verifier that supports your anchor rules—not the intermediate alone in a single PEM file for this client.

## 1. Decisions (recorded)

| Topic | Choice | Notes |
|-------|--------|--------|
| **CA / issuance model (plan §2, §9)** | **A — operational trust on the intermediate** | The **intermediate** CA signs **server** and other leaf certificates; the **root** only signs the intermediate. That matches a common split: day-to-day issuance uses the intermediate key; the root key stays on the **signing** side. This is the sense in which the plan calls **“policy A.”** |
| **Python MQTT client `ca-bundle.pem`** | **Root CA certificate** | Generated by `06-…` as a copy of **`certs/root-ca.crt`**, so the stock Python + OpenSSL TLS stack can **verify** the chain the broker sends (leaf + intermediate) up to a trust anchor. This is the **B**-style *client bundle* in the plan’s table (root in the file), while the **issuance** model above remains **A**. |
| **Server keys** | **ECDSA P-256** | End-entity server key and both CAs use **EC** `P-256` in these scripts. Change curves only with a clear compatibility review (Erlang/OpenSSL, corporate policy). |
| **SAN (default)** | **DNS: `localhost`** | `config/server-req.cnf` sets **Subject Alternative Name** to `localhost` so TLS hostname verification works when the MQTT client uses `host: "localhost"`. If you connect by IP or another DNS name, **edit the SAN (and your `host-params-*.json`) to match**—see [§4](#4-config-files-openssl-cnf). |
| **RabbitMQ** | **Docker + MQTTS** (e.g. **8883**) | Plain **1883** can remain for lab use; MQTTS is a **separate** listener. Exact wiring is in [§6](#6-rabbitmq-in-docker). |

A few things remain open globally (ciphers, minimum TLS version, whether to also **listen** for MQTT on 1883 in the same container): decide before production-like tests and encode them in `rabbitmq.conf` / environment as needed.

## 2. Directory layout (generated vs committed)

| Path | Role |
|------|--------|
| `local-broker/pki/config/*.cnf` | **Committed** — OpenSSL templates (DN, extensions, **SAN**). |
| `local-broker/pki/scripts/*.sh` | **Committed** — automation; run in order (see [§3](#3-script-run-order)). |
| `local-broker/pki/private/` | **Generated, gitignored** — private keys (`.key`). **Never commit.** |
| `local-broker/pki/certs/` | **Generated, gitignored** — issued certs, chain PEM, OpenSSL serial `.srl`. |
| `local-broker/pki/csr/` | **Generated, gitignored** — certificate signing requests. |
| `local-broker/truststore/ca-bundle.pem` | **Generated, gitignored** — **root CA** (`root-ca.crt` after step `06`); referenced by the MQTT client for TLS verification. |

Requires **OpenSSL** (1.1+ or 3.x) on the machine that runs the scripts (macOS: Apple CLT / Homebrew `openssl`).

## 3. Script run order

Run from the `local-broker/pki/scripts` directory (or invoke with absolute paths). Each script is idempotent in the sense that it **refuses to overwrite** existing key material; remove files only if you intend to re-issue everything.

| Step | Script | What it does |
|------|--------|--------------|
| 1 | `01-create-root-ca.sh` | Creates **root** private key and **self-signed root CA** certificate. |
| 2 | `02-create-intermediate-ca.sh` | Creates **intermediate** key + CSR, signs with **root** → **intermediate CA** cert. |
| 3 | `03-create-server-key-ecdsa.sh` | Creates the **RabbitMQ server (leaf) ECDSA** private key only. |
| 4 | `04-create-server-csr.sh` | Builds a **CSR** for the server using `config/server-req.cnf` (includes **SAN**). |
| 5 | `05-sign-server-csr-with-intermediate.sh` | Signs the server CSR with the **intermediate** (not the root) → **leaf** certificate. |
| 6 | `06-assemble-chain-and-truststore.sh` | Writes **`certs/broker-chain.pem`** (server + intermediate, **server first**) and **`../truststore/ca-bundle.pem`** (copy of **`root-ca.crt`** for the MQTT client). |

Equivalent to running steps 1–6 in order: **`run-all-pki.sh`** in the same `scripts/` directory (`bash run-all-pki.sh`). If any file already exists, the corresponding step exits with an error — use `99-clean-certs.sh` first to remove generated material.

**Summary (issuance + client verify)**

- **Root** (`root-ca.crt`, `root.key`): signs **only** the intermediate. **`ca-bundle.pem` = `root-ca.crt`** so Python/OpenSSL can anchor the server chain.
- **Intermediate** (`intermediate-ca.crt`, `intermediate.key`): issues **server** leaf certs. Included in **broker** chain, **not** alone in the client file (intermediate-only client bundles fail default PKIX verification here).
- **Server leaf** (`rabbitmq-server.crt` + `rabbitmq-server.key`): used by **RabbitMQ** for TLS. Presented with the **intermediate** in `broker-chain.pem`.

## 4. Config files (OpenSSL `.cnf`)

All live under `local-broker/pki/config/`.

| File | Purpose |
|------|---------|
| `root-ca.cnf` | `[req]` + `[v3_root]`: self-signed **root** (CA, `pathlen=1`, long-ish validity in script). |
| `intermediate-req.cnf` | CSR for the **intermediate** (subject DN only). |
| `intermediate-sign-v3.cnf` | Section `[v3_intermediate]`: CA constraints when **root signs** the intermediate (`pathlen=0`). |
| `server-req.cnf` | CSR and signed **server** cert: includes `[v3_server]` with **EKU serverAuth** and `subjectAltName = @alt_names`. |

**SAN (`[alt_names]` in `server-req.cnf`)**

- Default: **`DNS.1 = localhost`**. Aligned with MQTT `host: "localhost"` in `host-params-*.json`.
- If you add e.g. `IP.1 = 127.0.0.1` or another `DNS.2`, keep **the name the client uses for TCP** equal to a SAN entry, or verification fails.

**Distinguished names** (C, ST, L, O, OU, CN) in the `.cnf` files are **lab defaults**; adjust for realism or policy, but keep **one consistent stack** (root → intermediate → server) when you re-issue.

## 5. Optional checks

```bash
# Chain file order: end-entity first, then intermediate
openssl crl2pkcs7 -nocrl -certfile local-broker/pki/certs/broker-chain.pem 2>/dev/null | head
```

```bash
# Verify server cert against the same trust store the MQTT client uses
openssl verify -CAfile local-broker/truststore/ca-bundle.pem \
  -untrusted local-broker/pki/certs/intermediate-ca.crt \
  local-broker/pki/certs/rabbitmq-server.crt
# Expect: ... rabbitmq-server.crt: OK
```

**Optional (advanced):** treat the intermediate as a trust anchor in isolation (OpenSSL `partial_chain` — not what `mqtt_client.py` does):

```bash
openssl verify -CAfile local-broker/pki/certs/intermediate-ca.crt -partial_chain \
  local-broker/pki/certs/rabbitmq-server.crt
# Expect: ... rabbitmq-server.crt: OK
```

```bash
# Inspect SAN on the server cert
openssl x509 -in local-broker/pki/certs/rabbitmq-server.crt -noout -ext subjectAltName
```

## 6. RabbitMQ in Docker

After `06-assemble-chain-and-truststore.sh` you have:

- **Private key:** `local-broker/pki/private/rabbitmq-server.key`
- **Full chain (for listener):** `local-broker/pki/certs/broker-chain.pem` (server + intermediate; **order matters** for many TLS stacks)

### 6.1 What to configure

1. **Enable a TLS listener** (example port **8883**), distinct from plain MQTT on **1883** if you keep both.
2. **Point** `ssl_options` at the **chain PEM** and the **server private key** (see example snippet below). For **one-way** TLS (client verifies server, **no** client certificate), you typically **do not** need `verify_peer` for incoming MQTT clients; keep broker verification policy aligned with your team (often `verify_none` for the client side of the TCP connection when not using mTLS).
3. **Mount** the PEM files read-only into the container and reference **paths inside the container** in `rabbitmq.conf` (or the legacy `rabbitmq.config` if you use that style).
4. **MQTT over TLS** uses the same **SSL context** as other listeners in modern RabbitMQ when using unified listeners; ensure the **MQTT plugin** is enabled and clients target **8883** for MQTTS.
5. **Do not** copy **root** or **intermediate private keys** into the image; only the **server key** and **chain** for the listener.

### 6.2 Example `rabbitmq.conf` fragment (host paths are illustrative)

The repo’s **`docker/rabbitmq/mqtt-tls.conf`** and **`docker/rabbitmq/mqtt-plain.conf`** are the active copies mounted by **`docker-compose.yml`**; the example **`local-broker/pki/config/rabbitmq-tls.conf.example`** is the same idea with paths under `/etc/rabbitmq/tls/...` inside the container.

### 6.3 `docker-compose` adjustments (conceptual)

- Add **volume** mounts for `broker-chain.pem` and `rabbitmq-server.key`, and for a custom **`rabbitmq.conf`** that sets `listeners.ssl` / `ssl_options`.
- Expose **8883:8883** (or the port you choose) alongside **1883** if you use both.
- Rebuild or restart the container after changing certificates.

Exact compose snippets depend on your base image and whether you use `RABBITMQ_CONFIG_FILE` / `/etc/rabbitmq/rabbitmq.conf`; the RabbitMQ [TLS guide](https://www.rabbitmq.com/ssl.html) and [configuration](https://www.rabbitmq.com/configure.html) pages are authoritative for your version.

### 6.4 MQTT client (this repo)

- Set **`host`** in `host-params-*.json` to match **SAN** (default **`localhost`**).
- Use **port 8883** and enable TLS verification; set **`tls_ca_bundle`** to `truststore/ca-bundle.pem` (relative to `local-broker/`), which after step **`06`** is the **root CA** PEM (`root-ca.crt`).
